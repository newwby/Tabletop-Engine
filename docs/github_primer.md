# GitHub Primer for Project Contributors

This quick guide is for contributors who are new to GitHub. It covers what you need most: how to find project information, how to follow updates, and how to submit a useful issue when something needs attention.

## 1) What GitHub is (in plain language)
GitHub is a shared project workspace. Think of it as a mix of:
- a library (where project files live),
- a notice board (where updates are posted),
- and a request box (where issues are submitted).

You do **not** need to be a programmer to read project information or submit an issue.

## 2) Where to find what you need in this project
Start with these places:
- **README** (main project summary): what the project is and current direction.
- **/docs/** folder: project notes, decisions, and planning documents.
- **Issues** tab: open problems, requests, and discussion threads.

Tip: If you are unsure where to start, read README first, then check the Issues tab.

*[Image suggestion: Repository page highlighting README section and docs folder.]*

## 3) Reading updates without technical overload
To stay informed with minimum effort:
- Check issue titles first (they usually summarize the topic).
- Open only issues that sound relevant to you.
- Read the first post and latest comments before reading everything in between.
- Use reactions (👍) when you agree; this is useful feedback even without a long comment.

If wording feels technical, focus on: **What is the problem? Who is affected? What is being asked?**

## 4) How to open an issue (step by step)
1. Open the project repository on GitHub.
2. Click the **Issues** tab.
3. Click **New issue**.
4. Add a short, clear title.
5. Fill in the description (use the template if shown).
6. Submit the issue.

Before submitting, quickly check existing issues to avoid duplicates.

## 5) How to write a helpful issue
A good issue is simple and specific. Include:
- **What happened** (or what you need),
- **Where you saw it** (document/section/page),
- **Why it matters** (confusing, blocking, inconsistent, etc.),
- **What you expected instead** (if relevant).

Helpful structure:
- **Title:** one-line summary.
- **Details:** short paragraph or bullet points.
- **Optional:** screenshots, links, or copied text.

You do not need perfect wording. Clear and honest is enough.

## 6) What happens after you submit an issue
After submission, maintainers may:
- ask a follow-up question,
- add labels (for sorting and priority),
- link your issue to related work,
- close it when resolved or merged into another issue.

If someone asks for clarification, a short reply is usually enough. You are not expected to solve everything yourself.

## 7) Quick etiquette and confidence tips
- Be respectful and specific.
- One issue per topic is usually best.
- If you are uncertain, still submit the issue and say what you are unsure about.
- It is better to ask than stay blocked.

Your contribution is valuable, even if you only report confusion or ask for clarification.
